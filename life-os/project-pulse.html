02 / Diagnosis — Where the seconds live

Individual calls are fast.
The choreography is broken.

Every backend service hits its own p95 SLA — under 500ms. The problem isn't the services. It's that we wait for fifteen of them in a row, each one re-validating the same token, none of them cached, all of them ultimately reading from the mainframe.

Seven root causes — ranked by latency contribution

Bottleneck 01 · ~3.0s contribution

Sequential orchestration

The current aggregator awaits call N before firing N+1. Fifteen independent reads serialize into a six-second waterfall even when each one hits its SLA. No DataLoader, no parallel fan-out, no async/await fan-in pattern.

Bottleneck 02 · ~2.2s contribution

Mainframe hits the hot path

z/OS DB2 is on the read path for every page load. Positions, balances, settlements — all sourced live from the system of record with no read-through cache, no materialized view, no CQRS layer. Every refresh costs MIPS.

Bottleneck 03 · ~1.2s contribution

Synchronous risk recalc

VaR and exposure are computed at request time against a grid that's already under steady-state load. Treats every page view as a fresh risk run. Should be event-driven and cached, with on-demand recompute only when scenarios change.

Bottleneck 04 · ~600ms contribution

Auth re-validated per hop

JWT verification and entitlement lookups against IAM happen at every service boundary instead of at the gateway with a signed propagated context. Adds 80–200ms per call, compounded across the fan-out.

Bottleneck 05 · ~500ms contribution

Payload bloat

REST endpoints over-fetch. A 4 KB positions object delivered when the tile only needs symbol, qty, mark, P&L. No GraphQL field selection, no projection at source, just JSON dumped over the wire.

Bottleneck 06 · ~540ms contribution

Render blocked on slowest tile

The page waits for the last call to land before painting. The slowest tile (almost always risk or settlements) sets the user-perceived load time. No skeleton screens, no streamed SSR, no @defer.

Bottleneck 07 · cross-cutting

No observability into the right things

Splunk catches errors. AppDynamics catches slow endpoints. Neither catches the perceived load time of the page from the user's device — the only number that matters. Without RUM (Real User Monitoring) we're tuning what we can measure, not what users feel.

03 / Solution — Five compounding levers

No single fix gets us to two seconds.
Five layered ones do.

Each lever is independently deployable, independently measurable, and independently revertible. That matters because the brief mandates incremental rollout and no full rewrite. The five compose multiplicatively: parallel + cache + event-driven + read-replica + progressive render takes a 7.4s page to 1.7s.

01
GraphQL BFF for parallel fan-out
Apollo / DGS DataLoader
A single accountOverview query at the BFF resolves all dependencies in parallel via DataLoader-style batching. Fifteen sequential 400 ms calls collapse to one fan-out capped by the slowest dependency — typically under 700ms. The BFF also handles field projection, so payloads shrink to what the tile actually needs. Same pattern NAB, Capital One, and Wealthsimple use today; same pattern Citi Velocity 3.0 uses for micro-frontends.
02
Multi-tier caching with explicit TTLs
Hazelcast Gemfire Akamai
Edge CDN (Akamai) for static fragments and market data widgets — TTL 30–60s. Hazelcast/Gemfire grid for hot account state — positions, balances, P&L — TTL 5–15s with event-driven invalidation. In-process Caffeine cache at the BFF for ultra-hot entitlements — TTL 60s. Apollo Client cache on the device for instant navigation back. Every tier has an explicit freshness contract per data type.
03
Event-driven cache warming via CDC
Kafka Solace Debezium
Debezium emits change-data-capture from DB2 and Oracle. Kafka (or Solace PubSub+, already deployed for trade messaging) carries the events. Cache writers consume and update Hazelcast ahead of the next user request. The data becomes fresh-when-pushed, not fresh-when-pulled. This single lever decouples user latency from system-of-record latency.
04
CQRS — read store separate from write store
PostgreSQL ClickHouse Elasticsearch
Mainframe stays authoritative for writes. A purpose-built read store holds denormalized account snapshots refreshed by the event stream: PostgreSQL for transactional reads, ClickHouse for analytical tiles (intraday P&L curves, exposure-by-counterparty), Elasticsearch for trade and document search. Reads short-circuit the legacy path entirely. Mainframe MIPS savings of 15–25% on read traffic.
05
Progressive UI rendering — skeleton + stream
React Suspense @defer / @stream SSE
Skeleton paints in under 200ms. Tiles stream in as data arrives — positions first (cached, instant), then trades, then risk, then offers. GraphQL @defer and @stream directives handle the chunked transport. Server-Sent Events push live updates after first render. This is the lever that gets us to perceived sub-2s even when the slowest tile takes three.

04 / Architecture — End to end

What the target state actually looks like

Five tiers, three caching layers, two data paths — hot (Hazelcast hit, sub-100ms) and cold (microservice + read replica, sub-700ms). The mainframe sits below, no longer on the user's critical path.

HOT PATH · cached

~120ms p95

Client → CDN/Gateway → BFF → Hazelcast hit → BFF returns. No mainframe touch. Covers ~70% of session traffic after warmup.

COLD PATH · cache miss

~650ms p95

Client → BFF → microservice → read replica → response written to Hazelcast → BFF returns. Mainframe only for first-of-day reads.

LIVE PATH · streamed

~50ms push

Event in Kafka → WebSocket fan-out to subscribed clients. Position updates, trade confirms, risk alerts arrive without polling.

The example query — what the client actually asks for

# GraphQL query issued by the client on Account Overview mount.
# Single round-trip · field selection drives projection at source.

query AccountOverview($clientId: ID!) {
  client(id: $clientId) {
    id
    displayName
    advisor { name avatarUrl }

    balances {
      currency
      available
      pending
      @cached(ttl: 10)
    }

    positions(top: 20) {
      symbol
      assetClass
      quantity
      marketValue
      pnl { intraday since: "openOfBusiness" }
      @stream(initialCount: 10)
    }

    recentTrades(last: 5) {
      tradeId tradeDate symbol side qty price status
      @defer(label: "trades")
    }

    risk {
      var95 var99 stress
      exposure { byCounterparty(top: 5) { name amount } }
      @defer(label: "risk")
    }

    settlements(status: ALL) {
      tradeId expectedDate actualDate status
      @defer(label: "settle")
    }

    aiInsights {
      # R2D2-backed agent: produces 1-3 contextual insights
      # RAG over: account history, market news, compliance events
      headline body action confidence
      @defer(label: "ai")
    }
  }
}

The @stream directive tells the BFF to begin returning positions as soon as the first 10 are ready, then push the rest. The @defer directive marks tiles that can paint after the initial response — risk, settlements, AI insights. The client sees structure in 200ms, real data in 600ms, full page in 1.7s. One query, one connection, one mental model.

07 / AI Layer — R2D2, RAG, and Agents

The intelligence sidecar.
Not the system of record.

Pulse treats AI as a distinct concern from data delivery — an enrichment sidecar, not a replacement for deterministic services. The page renders correctly without AI. AI makes it useful. That separation is critical for regulatory defensibility: every number on the screen has a deterministic source; every insight is generated and clearly tagged as such.

AI-01 · Developer productivity

R2D2 in the inner loop

Citi's R2D2 already accelerates development across ICG teams. For Pulse, R2D2 helps generate GraphQL resolvers from REST API specs, scaffolds DataLoader batching boilerplate, suggests cache TTL policies based on domain semantics, and reviews PRs for compliance with the BFF pattern. Estimated 25–35% reduction in BFF onboarding time per service.

AI-02 · RAG-powered insights

Smart Account Summary tile

A retrieval-augmented LLM reads the user's account state (positions, recent trades, settlements) and overlays it with vector-retrieved context: relevant compliance updates, market news, internal research notes, regulatory filings. Produces a 2-3 sentence "what changed since you last logged in" headline. Vector store: pgvector on PostgreSQL or Pinecone for hosted. Every cited source is linkable and audit-logged.

AI-03 · Agentic offers

Personalized advisory feed

An agentic pipeline runs nightly per client portfolio: identifies asset-allocation drift vs. mandate, scans for tax-loss harvesting opportunities, flags cash drag on uninvested balances, surfaces relevant Citi products. Top-3 actionable offers land in the overview as discrete cards with clear "why this" rationales. Compliance-reviewed prompts; deterministic guardrails on what can be suggested.

AI-04 · Natural language query

"Ask Pulse"

A scoped LLM gateway converts natural language ("show me my top exposures to European banks", "what trades settled late this week") into safe GraphQL queries against the user's own data scope. Translation happens in a sandboxed agent; results render in the same tiles. Query intent is logged for audit; query execution uses the same entitlements as any other request.

AI-05 · Anomaly + compliance

Background watchdog agent

A passive agent monitors the position + trade event stream for anomalies: unusual position size, off-pattern counterparty, missed margin call, regulatory threshold breach (MiFID II, EMIR, Dodd-Frank). Alerts route to the regulatory tile with explanation and recommended next step. Catches issues earlier than nightly batch surveillance.

AI-06 · LLM gateway and guardrails

Centralized control plane

All LLM calls flow through an internal LLM gateway that enforces: model choice (no PHI/PII to external providers), prompt logging, output filtering, cost attribution per team, and rate limiting. PII never leaves the perimeter for the BFF use cases. External model access reserved for non-customer-data agent calls.

How AI fits the latency budget

AI tiles are always deferred. They never block the page paint. The deterministic data (positions, balances, P&L) arrives in under a second. The AI sidecar typically returns insights in 600-1200ms — comfortably within the 1.7s total budget when streamed via @defer. If the LLM is degraded or rate-limited, the page still renders correctly — the insight tile simply shows a graceful empty state. AI augments. It never gates. 

# AI insight resolver — runs in parallel, deferred from page paint
@Component
@DgsComponent
public class AiInsightResolver {

    @DgsData(parentType = "Client", field = "aiInsights")
    public CompletableFuture<List<Insight>> resolveInsights(DgsDataFetchingEnvironment dfe) {
        Client client = dfe.getSource();

        // Step 1: gather context from cache (no slow calls inline)
        AccountContext ctx = contextCache.get(client.getId());

        // Step 2: fire RAG retrieval + LLM in parallel, with timeout
        return CompletableFuture.supplyAsync(() ->
            llmGateway.generateInsights(
                ctx,
                ragService.retrieve(client.getId(), topK: 5),
                Duration.ofMillis(1200)   // hard timeout
            )
        ).exceptionally(ex -> List.of());  // graceful empty
    }
}

09 / Data — Sharding, indexing, caching

The new
data topology

Performance at IB scale is a database problem before it's anything else. Pulse introduces deliberate sharding, targeted indexing, and a three-tier cache hierarchy. The mainframe stays the system of record — but it stops being on the read path.

Three-tier cache hierarchy

L1 · In-process · < 1ms

Caffeine

What it holds. Entitlements, user preferences, schema metadata, rate-card lookups.

TTL. 60 seconds, refresh-ahead on near-expiry.

Why. Hot lookups on every request. Nanosecond access. Invalidated by Kafka entitlement events.

L2 · Distributed · < 10ms

Hazelcast / Gemfire

What it holds. Positions, balances, P&L, recent trades, FX rates, risk snapshots.

TTL. 5–15s per data type, with CDC-driven invalidation overriding TTL on changes.

Why. Shared across BFF instances; near-cache on each instance for ultra-hot keys. Already operationally familiar at Citi.

L3 · Edge · < 50ms

Akamai + CloudFront

What it holds. Static UI assets, public market data, non-personalized offer feeds, reference data.

TTL. 30s for market data, 1hr for offers, immutable for hashed assets.

Why. Closer to user than any origin. Eliminates round-trip for non-personalized data.

Sharding strategy for the read store

Store Shard key Why this key, why this store
PostgreSQL — positions, balances hash(client_id) % N All of a client's positions and balances co-located on the same shard. Page-load queries are per-client, so a single shard answers. Re-sharding via consistent hashing; N = 32 to start.
PostgreSQL — trades range(trade_date) Time-range queries dominate. Recent trades (last 7d) hot. Older trades cold and on separate hardware. Partition pruning is automatic.
ClickHouse — P&L time series (client_id, date) Columnar; queries are aggregate over time windows per client. Sort key is the same as shard key. Intraday tick-level retained for 30 days, daily aggregate for 5 years.
Elasticsearch — trade search routing = client_id All client docs in one shard for fast per-client retrieval. Cross-client search uses standard distribution; rare path so the cost is acceptable.
Hazelcast — hot grid partitioner(client_id) Custom partitioner ensures all of a client's hot state lives on the same Hazelcast member. Affinity routing — BFF instance pinned to nearest member.
pgvector — RAG embeddings hash(corpus_id) Corpora are domain-bounded (compliance, market research, internal notes). Cross-corpus retrieval rare; per-corpus retrieval routine.

Indexing — what we add, and why

POSITIONS TABLE

Composite (client_id, asset_class, symbol)

Drives the default Overview query: "all positions for client X, grouped by asset class". Covering index includes market_value and quantity to make this an index-only scan.

TRADES TABLE

Composite (client_id, trade_date DESC) + GIN on metadata

BTree handles recent-trades queries; GIN handles JSONB attribute search ("trades with strategy=DCA"). Partial index on status='settled' reduces hot rows.

SETTLEMENTS TABLE

Partial index on status IN ('pending','failed')

The vast majority of historical rows are 'settled' and never queried. Indexing only active states keeps the index 50× smaller and dashboard queries 10× faster.

VECTOR STORE

HNSW index on embedding column

Approximate nearest neighbor search at low millisecond latency. m=16, ef_construction=64 balances recall vs. memory. Reindex job runs nightly on corpus refresh.

# Read-side: BFF resolves a field by consulting tiers in order.
# Hazelcast hit (70%+) returns in ~10ms. Miss falls through cheaply.

function resolvePositions(clientId) {
  return caffeine.get(clientId)                // L1 nanos
      ?? hazelcast.get(clientId)               // L2 ~10ms
      ?? postgres.query(clientId)              // L3 ~150ms
            .also(res -> hazelcast.put(clientId, res));
}

GraphQL N+1 / query cost blowup Low Medium DataLoader enforced by lint rule. Query complexity analysis at gateway · max depth and cost limits. Persisted queries only — no ad-hoc queries from prod clients.
Event ordering / out-of-order updates Medium Medium Kafka partition key = client_id ensures per-client ordering. Event versioning at consumer. Last-write-wins with timestamp tiebreak. Periodic reconciliation against SOR catches drift.
LLM hallucination / wrong insight High Medium Insight tile is always advisory · never decisional. Every numeric claim must cite a deterministic source. Human review on offer prompts. Strict RAG groundedness checks. Compliance pre-clears all prompt templates.
Org change resistance Medium Medium Each phase is a clean win that doesn't require Phase N+1 buy-in. RUM data is shown weekly so improvements are visible. Service teams onboarded gradually with R2D2 scaffolding to reduce friction.
Cost overrun Low Medium Stack is mostly already paid for (OpenShift, Hazelcast, Gemfire, Solace, Splunk). New costs (Kafka, ClickHouse, RUM, vector store) modeled at ~$2-4M/year against the ~$60M+ in productivity + MIPS savings.
Security regression Low Critical Auth + entitlements remain in gateway and microservices, not BFF. Field-level authorization in GraphQL schema for sensitive fields. Penetration test before each phase. ICG InfoSec on the design review.

The tradeoffs we accept openly

Speed Freshness

Slight lean toward speed for non-trading data (offers, market widgets — 30-60s TTL is fine). Trading-critical data (positions, P&L, risk) gets sub-15s freshness via event-driven invalidation. Per-tile contract.

CX impact Implementation effort

Heavy lean toward CX. Phase 1 quick wins (skeleton + parallel + cache) deliver ~50% of total improvement for ~10% of effort. That asymmetry is intentional and front-loaded.

Scalability Cost

Slight lean toward scale. New cost is small ($2-4M/yr); savings are large ($60M+ over 18mo). When marginal cost is dominated by mainframe MIPS, scaling out cache + read replicas is the cheap direction.

Optimization Architectural complexity

Centered. We accept moderate new complexity (CQRS, eventing, BFF) because each piece is independently understood and operated. We do NOT accept exotic complexity — no service mesh-of-meshes, no premature Apollo Federation, no microservices-per-function.

Security Accessibility

Bias toward security. Auth, entitlements, PII handling remain in the existing hardened layers. The BFF resolves through them, not around them. Accessibility (sub-2s, mobile-first, WCAG 2.1 AA) is achieved within that posture — the brief explicitly requires it.
