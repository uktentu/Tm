01   The shape of the problem

Fifteen fast calls
do not make a fast page.

Individual backend APIs respond in under 500ms. Yet the page takes five to ten seconds. The arithmetic is obvious once seen: sequential orchestration, repeated authentication, uncached mainframe reads, and synchronous risk recalculation stack up into a waterfall that no single team owns and no single fix resolves.

The Account Overview surface in Citi’s investment banking context is not a retail balances page. It assembles positions across legal entities, intraday P&L, exposure metrics, recent trades and confirmations, settlement status, FX and cash balances, market data widgets, personalized offers, and increasingly an AI-curated “next best action” tile. Behind these tiles sit twenty or more dependencies spanning microservices on OpenShift, mainframe systems of record on z/OS DB2, middle-office trade stores, risk engines, and external feeds.

The five-to-ten-second symptom decomposes cleanly into six contributors. Each is small on its own. Together they are the page.

Diagnosis

The page is not slow because any one call is slow. It is slow because nothing is done in parallel, nothing is cached at the edge, the mainframe sits on the critical path, and the browser paints nothing until the slowest tile resolves.

03   The solution, in five levers

Five moves
that compound.

Each lever is independently deployable and independently measurable. None of them require a platform rewrite. The constraint says incremental rollout preferred — this is what incremental looks like when designed deliberately.

01

GraphQL BFF for parallel orchestration

A single accountOverview query fans out to all dependencies in parallel via DataLoader-style batching. Fifteen sequential 400ms calls collapse into one parallel batch capped by the slowest individual call.

Apollo Federation or Netflix DGS on the Java side keeps the BFF inside the existing service-mesh, OpenShift, and Splunk ecosystems.

Owns: speed · payload shape
02

Multi-tier caching with explicit TTL contracts

CDN edge for static and market data. Hazelcast or Gemfire for hot session and position data with sub-15-second TTL. Caffeine for in-process L1. Apollo Client cache for the browser.

Each tier has a written staleness contract per data type. Freshness is a product decision, not an accident.

Owns: server load · mainframe MIPS
03

Event-driven cache warming via CDC

Kafka or Solace consumes change-data-capture from DB2 and trade stores. Cache writers update Hazelcast ahead of the user’s next request. Positions become fresh-when-pushed, not fresh-when-pulled.

This is the lever that resolves the speed-versus-freshness tension. Fresh data arrives by event, not by blocking read.

Owns: freshness · staleness windows
04

CQRS read replicas and materialized views

A purpose-built read store — PostgreSQL with logical replication for transactional reads, ClickHouse for analytical tiles like intraday P&L curves — holds denormalized account snapshots refreshed by the event stream.

Mainframe remains authoritative for writes. Reads short-circuit the legacy path entirely.

Owns: mainframe offload · analytics
05

Progressive rendering and streaming UI

The page paints a skeleton in under 200ms. Tiles stream in as data arrives via GraphQL @defer and @stream directives or server-sent events. Positions appear first, then trades, then risk, then offers.

This is what gets perceived load under two seconds, even when a heavy analytical tile genuinely takes three.

Owns: perceived performance
06

AI-driven prefetch and personalization

An LLM-backed predictor — running on Citi’s R2D2 platform or an analogous internal service — observes the user’s pre-login signal and pre-warms the cache for tiles they are statistically likely to request first. For a trader on the FX desk at 7am London, that is positions and overnight P&L. For a private banker, that is client roster and recent transfers.

Owns: personalized speed
04   Target architecture

The fast path,
drawn out.

Same five levers in their final composition. The browser talks to a thin gateway, the gateway talks to a BFF, and the BFF answers most queries from cache before ever touching a backend service or the mainframe.

Target state · multi-tier cached, event-warmed Target latency < 2,000 ms · p95

The mainframe still exists. It still owns the truth. But it no longer participates in the synchronous read path — every read happens against a cache or a replica, refreshed by events. Mainframe MIPS go to writes and to settlement, where they belong.

06   Cloud-native deployment

OpenShift, AWS, GCP —
and the hybrid reality.

Citi operates a hybrid model. Core systems of record stay on-prem. New-build services live on AWS and GCP. The Account Overview spans both. The architecture has to acknowledge that on day one, not as a future migration.

Compute and orchestration

The BFF runs on OpenShift in the existing on-prem fabric for proximity to authentication and entitlement services. Stateless tile services run on AWS EKS or GCP GKE where elasticity matters more than proximity. Service mesh (Istio) carries identity across the boundary using SPIFFE / SPIRE.

Deployment pipelines move from legacy Jenkins to Tekton on OpenShift for cloud-native deploys, with Harness as the orchestration layer above. Lightspeed handles the actual rollout to production. Code lives in GitHub Enterprise.

Storage and data

Mainframe DB2 stays. Aurora PostgreSQL (AWS) or AlloyDB (GCP) becomes the read replica home, with logical replication driven by Debezium CDC. ClickHouse or BigQuery handles analytical tiles — intraday P&L curves, exposure-by-counterparty heatmaps.

Object storage for large blobs (statements, confirms, regulatory exports) sits on S3 with cross-region replication. KMS handles encryption keys at rest. The boundary between on-prem and cloud is the Kafka event stream, which is consumed by both.

Database sharding and indexing strategy

Account Overview reads are bounded by legal entity, by customer ID, and by trade date. That makes the natural shard key a composite of legal entity and customer ID hash. Range partitioning on trade date sits underneath, which keeps recent activity hot and old activity cold without complicating reads.

07   Event-driven architecture

Eventing turns the
tradeoff into a non-issue.

The classic objection to caching in a bank is staleness. The classic answer is short TTLs, which only mitigates the problem. The actual answer is that the cache should be invalidated by the event that changed the data, not by a timer that hopes it has.

Every write to a system of record — a trade booked, a settlement confirmed, a position rolled, a wire received — produces a Kafka event. Cache writers subscribe to those events and update Hazelcast or Gemfire within milliseconds of the underlying change. The BFF reads from cache. The user sees fresh data.

The event stream is also the broadcast channel to the client. The same Kafka topic that warms the cache also fans out via WebSocket or SSE to every connected session that holds a subscription on that customer or that trade. A settlement confirmation lands as a push, not a poll. A new fill appears on the blotter the moment the exchange acknowledges it.

The freshness contract

Static data and offers: 30 to 60 seconds via CDN. Balances and positions: 5 to 15 seconds via Hazelcast with event invalidation. Active orders, fills, risk for traders with open positions: pushed via WebSocket, latency measured in single-digit milliseconds.

08   Server-side rendering and progressive UI

Paint the page
before the data arrives.

A modern bank’s front-end stack can render a useful, branded skeleton in two hundred milliseconds. That is the move that converts a five-second wait into a sub-two-second perception.

The render strategy

The shell of the page — navigation, layout, tile placeholders, the user’s name, the date, the legal entity selector — is server-rendered on the edge using a thin Next.js or SvelteKit layer. It ships pre-rendered HTML in the first byte.

Tile data arrives in subsequent streamed chunks via React Server Components, GraphQL @defer directives, or SSE. Each tile owns its loading state, error boundary, and skeleton.

The micro-frontend boundary

Citi’s Velocity 3.0 already proved this pattern in FX trading: micro-frontends let tile teams ship independently. The Account Overview becomes a composition of tiles — positions, P&L, trades, offers, alerts — each owned by the team that owns the underlying domain.

Module Federation handles runtime composition. Each tile is versioned independently and rolls out behind its own feature flag.

Why this clears the < 2s bar

First contentful paint at ~200ms. Largest contentful paint — the positions tile, typically the most important — at ~800ms because positions live in Hazelcast hot cache. Trades and balances stream in at ~1.2s. Heavy analytical tiles (risk decomposition, exposure-by-counterparty) stream in by 1.8s. Slow tiles arrive after 2s but the user already has a usable page.

09   AI, LLMs, agents

R2D2, RAG,
and the predictive cache.

The most interesting part of this proposal is also the newest. Citi’s internal AI platform — R2D2 and the agent infrastructure being stood up around it — is not just a feature surface on the Account Overview. It is a performance lever. The cache that is warmed by an LLM prediction does not have a freshness problem.

P

Predictive prefetch

A lightweight model trained on session telemetry predicts the user’s next three tiles before they request them. The BFF pre-warms cache entries during the auth round-trip. The tiles are already in Hazelcast by the time the GraphQL query arrives.

Lever: speed
R

RAG for the inline assistant

An LLM with retrieval over the user’s positions, recent trades, and policy documents answers questions inline. The retrieval index is built off the same Kafka stream that warms the cache — one event, two consumers, no extra ETL.

Lever: surface area
A

Agentic next-best-action

A planning agent observes context (market open, user role, recent activity) and queues actions on the Account Overview: a rebalance suggestion for a portfolio manager, a settlement chase for an operations user, a confirm-pending nudge for a trader.

Lever: value density
D

Anomaly-driven cache eviction

An LLM-classified anomaly stream — built on R2D2 — identifies when a position or balance has moved enough that the cache should be flushed immediately rather than waiting for the next CDC event. The cache becomes self-correcting under stress.

Lever: correctness
Q

Query-shape optimization

The BFF logs every GraphQL query shape. A nightly LLM analysis identifies query patterns that are slow or N+1 and proposes either resolver-level changes or materialized-view refreshes. Performance engineering becomes a continuous, AI-assisted loop.

Lever: engineering velocity
G

Governed by policy and audit

Every agent action is logged via the same audit pipeline that handles trade booking. Every RAG retrieval is policy-scoped against the user’s entitlements before the LLM sees a token. The audit story is the same one MiFID II already accepts.

Lever: compliance posture
Why this matters for the page-load problem

Three of the six AI modules above are direct performance levers. Predictive prefetch removes the cache miss. Anomaly-driven eviction removes the staleness window. Query-shape optimization removes the slow query before it ever lands in production. AI is not a feature bolted onto the architecture — it is the architecture’s control loop.

12   Precedent

Banks that
did this already.

Nothing in this proposal is speculative. The pattern set has been deployed at scale across multiple banks. The most relevant precedent is Citi’s own.

Citi Velocity 3.0FX single-dealer platform

Citi consolidated its FX trading platforms in 2023 onto a single HTML5, micro-frontend, cloud-native dealer platform. The same architectural pattern set this proposal advocates — HTML5 streaming libraries, micro-frontends, cloud-native infrastructure — was already applied successfully to FX trading.

50% faster cold start · 30% faster hot start · 71% faster deal booking · 94% faster order placement response · 65% faster blotter load

This is the strongest internal benchmark. The proposal here is to extend the established blueprint to one more surface — the Account Overview — with the addition of the BFF, CDC, and AI layers.

CitiDirectCash and treasury platform

Citi’s broader transformation of CitiDirect from a tightly coupled legacy framework to a microservices-based modular structure is the institutional precedent for moving off batch processing and point-to-point integrations. The Account Overview proposal is a vertical slice of that same transformation philosophy.

National Australia BankNAB-X · GraphQL BFF

NAB runs GraphQL in a BFF topology where each miniapp frontend has a paired GraphQL server. Critically, they resolved data through downstream REST APIs rather than directly to data sources — preserving the security, authentication, and entitlement controls already enforced in the existing API gateway. That is the pattern that satisfies the security-versus-accessibility constraint in this brief without a re-do of auth.

WealthsimpleHive · centralized GraphQL gateway

Wealthsimple built a centralized GraphQL gateway on TypeScript and Node.js as an intelligent router orchestrating requests across the service landscape, using schema stitching to unify microservice schemas. Their adoption validates the federation-ready posture this proposal recommends — start with a single BFF, federate when 3+ teams own subgraphs.

Capital OneGraphQL adoption

Capital One’s public analysis frames why BFF and GraphQL win in mobile banking: a single customized endpoint aggregates data from multiple microservices, filters what is needed for the channel, and returns only the necessary data. That is the same payload-shape argument that delivers the second half of the latency reduction in this proposal.

14   Tradeoffs

Where this proposal
does not pretend.

Every tradeoff in the brief has a specific resolution. None of them are hand-waved.

TradeoffResolution
Speed vs data freshness Resolved by tiering. Each data domain has a written freshness contract. Market data tolerates 30 — 60s via CDN. Balances and positions tolerate 5 — 15s via Hazelcast with event invalidation. Active orders, fills, and risk for traders with open positions get push semantics via WebSocket at millisecond latency. The freshness decision is per-domain, not per-page.
Customer experience vs implementation effort Skeleton plus progressive rendering is the disproportionate first win. Two weeks of frontend work for a 30 — 50% perceived improvement. Earn the cheap win first, then justify the BFF investment with measured impact and the case study to back it.
Scalability vs cost Hazelcast at IB volumes is roughly $0.2 — 0.4M annual incremental for a major segment. CDN cost is offset by reduced backend QPS. The real saving is mainframe MIPS reduction — every cached read is a DB2 transaction not billed. Typical break-even is 18 months. Kafka for CDC is the largest new cost and the largest strategic lever.
Optimization vs architectural complexity Apollo Federation introduces real complexity — schema governance, distributed tracing, query cost analysis. The mitigation is to start with a single non-federated BFF and only federate when three or more teams need to own subgraphs. Premature federation killed projects at other banks. NAB explicitly hasn’t done it.
Security vs accessibility The NAB pattern: keep authentication and entitlements in the existing API gateway and microservices. The BFF resolves data through those gateways, never around them. Add field-level authorization on the GraphQL schema so a misconfigured query cannot exfiltrate. Mask PII at the BFF, not the client. No new auth surface; reduced auth surface.

For a single FX desk doing two hundred logins per day across fifty traders, dropping five seconds to one and a half saves approximately thirty hours of cumulative wait per week. At fully-loaded investment-banking labor cost, that is in the order of $1.5M per desk per year. Multiply by the number of desks. The business case is not the page load. 16   The catalog

Every bottleneck,
named.

A complete enumeration of the contributors to the five-to-ten-second observed latency. Each one is independently addressable. The right column is the lever from section three that retires it.

01

Sequential API orchestration (the waterfall)

The aggregator awaits call N before firing call N+1. Fifteen calls at 400ms each serialize to six seconds. By a wide margin the largest single contributor.

Lever 1 · GraphQL BFF with DataLoader-style batching turns the waterfall into a single parallel batch.

+ 3,000 ms
Orchestration
02

Repeated authentication and entitlement checks

JWT or SAML re-validation and per-service entitlement lookups against an IAM mainframe add 80 — 200ms per service hop. Fifteen hops compounds dramatically.

JWT verification cached at the gateway; entitlements pre-resolved once per session at the BFF, then attached as a signed context to downstream calls.

+ 600 ms
Identity
03

Uncached mainframe DB2 reads

Every page load hits z/OS DB2 for positions and settlements. No edge cache. No materialized view. Each read is 200 — 800ms and billed in MIPS.

Hazelcast hot cache plus PostgreSQL read replica refreshed by Debezium CDC. Mainframe leaves the synchronous read path.

+ 2,200 ms
Data
04

Synchronous risk and P&L recalculation

VaR, exposure, and P&L recomputed at request time instead of being pre-aggregated. The most computationally expensive call on the page also has the highest tail latency.

Pre-aggregated snapshots in ClickHouse refreshed by the trade event stream. On-demand recalc reserved for explicit user request.

+ 1,200 ms
Compute
05

Over-fetching and payload bloat

REST endpoints return 4KB position objects when the page tile only needs symbol, quantity, and P&L. Multiplied across fifteen calls, payload weight blocks parsing and rendering.

GraphQL queries request exactly the fields rendered. Payloads drop 60 — 80% in most cases.

+ 200 ms
Payload
06

Render blocking on the slowest tile

The browser paints nothing until the slowest backend resolves. Risk decomposition takes three seconds — the page is blank for three seconds.

Server-side rendered skeleton in 200ms. Tiles stream in via @defer or SSE, each with its own loading state. The page feels usable before the data lands.

+ 500 ms perceived
Frontend
07

Cross-region data hops

An NY trader pulling a London-booked position incurs trans-Atlantic round-trips per dependency. RTT alone adds 80 — 100ms per call.

Regional read replicas for hot data. Active-active topology where regulatory permits. The data is closer to the user than the user is to the data.

+ 400 ms
Network
08

N+1 query patterns in trade history

A list of twenty trades triggers twenty individual lookups for related entities (counterparty, currency pair, allocation). Classic ORM antipattern.

DataLoader batching at the BFF, plus denormalized read views in the read replica that pre-join the related entities.

+ 400 ms
Data access
09

Inefficient indexes on transaction tables

Trade and settlement tables indexed for write paths, not read paths. Queries filtered by customer ID and date scan more rows than they return.

Read-path-specific covering indexes on the replica. BRIN indexes for cold date ranges. The write path on the mainframe is untouched.

+ 300 ms
Data
10

JSON parsing overhead at scale

Verbose REST JSON serialization and deserialization at every hop. CPU-bound, not network-bound. Compounds across the mesh.

Protocol Buffers or Avro on inter-service hops. GraphQL persisted queries to remove the parse cost on the gateway entirely.

+ 150 ms
Serialization
11

SSL/TLS handshake overhead

mTLS in the service mesh is mandatory and correct. But every short-lived call pays the handshake cost. Connection pooling is often misconfigured.

Persistent HTTP/2 connection pools with proper idle timeouts. TLS session resumption. Istio defaults audited and tuned.

+ 200 ms
Network
12

Database connection pool exhaustion at peak

Login storms at market open exhaust connection pools. New connection acquisition adds hundreds of milliseconds to the tail.

PgBouncer or Vitess-style connection multiplexing. Cache-first strategy means most reads never reach the pool. Capacity planning against peak login arrival.

+ 400 ms (tail)
Infrastructure
13

JVM garbage collection pauses

Long-running Java services with large heaps experience 100 — 500ms stop-the-world pauses. Visible in the p99 latency distribution.

ZGC or Shenandoah on the BFF. Heap sizing reviewed. Tail-latency-sensitive paths split into dedicated services with smaller heaps.

+ 200 ms (p99)
Runtime
14

Log aggregation and synchronous instrumentation

Verbose synchronous logging to Splunk on the hot path. AppDynamics agents sampling at the wrong rate. Observability that costs more than it returns.

Async log appenders. OpenTelemetry sampling tuned per route. Critical-path observability split from diagnostic observability.

+ 80 ms
Observability
15

Cold caches at peak login times

Market-open login storms hit cold Hazelcast nodes that have rotated keys overnight. Cache miss avalanche.

Scheduled cache warming starting 30 minutes before market open, driven off the user’s last-known-good footprint. AI-driven prefetch for the heavy users.

+ 700 ms (peak)
Cache
16

Mobile network variability

Mobile clients on cellular networks see higher and more variable RTT. The fast-path architecture matters more on mobile, not less.

HTTP/3 (QUIC) where supported. Aggressive CDN edge caching. Mobile-specific GraphQL queries that return less. Service worker offline-first for the shell.

+ 600 ms (mobile)
Client
17

Third-party feed timeouts on the critical path

External feeds — Bloomberg, Refinitiv, SWIFT GPI — occasionally degrade. When their tile is on the synchronous render path, the entire page degrades with them.

Hystrix or Resilience4j circuit breakers. Stale-while-revalidate cache strategy. Degraded tile fails open with last-known-good data and a discrete indicator.

+ 1,000 ms (degraded)
External
18

No client-side caching across navigations

Navigating away from the Account Overview and back triggers a full reload. Browser history, Apollo Client cache, and service worker are all underused.

Apollo Client normalized cache with type policies per domain. Service worker for the shell. Back/forward navigation is instant.

+ 2,000 ms (return visit)
